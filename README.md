Тест кейси для ендпоінта **POST /store/order**

**1. Успішне створення замовлення з валідними даними**

**STR:**
- Надіслати POST /store/order з валідними orderId, petId, quantity (1–5), shipDate (ISO), status="placed", complete=false

**ER:** HTTP 200. Замовлення створене. У відповіді id, petId, quantity відповідають переданим значенням



**2. Створення замовлення з неіснуючим petId**

**STR:**
- Надіслати POST /store/order з валідними даними, але з неіснуючим petId

**ER:** Очікувано 4xx (400	Invalid Order - по документації). Якщо 200 — відсутня перевірка референсної цілісності



**3. Повторне створення замовлення з тим самим orderId**

**STR:**
- Надіслати POST з orderId = X
- Повторити POST з тим самим orderId = X

**ER:** 4xx або перезапис/дублікат (очікувану поведінку краще уточнити)



**4. Перевірка поля status на невалідне значення**

**STR:**
- Надіслати POST з status="invalid_status"

**ER:** 4xx



**5. Відсутність поля orderId**

**STR:**
- Надіслати POST без поля orderId

**ER:** 4xx або автогенерування значення



**6. Відсутність поля shipDate**

**STR:**
- Надіслати POST без поля shipDate

**ER:** 4xx або автогенерування значення



**7. Створення замовлення з мінімальним quantity**

**STR:**
- Надіслати POST з quantity = 1
  
**ER:** HTTP 200. quantity у відповіді = 1



**8. Створення замовлення з великим quantity**

**STR:**
- Надіслати POST з дуже великим quantity (наприклад, 999999)

**ER:** HTTP 200. quantity у відповіді = 999999



**9. Створення замовлення з quantity = 0**

**STR:**
- Надіслати POST без поля shipDate

**ER:** Очікувано 4xx. Якщо 200 — дефект: відсутня валідація



**10. Створення замовлення з негативним quantity**

**STR:**
- Надіслати POST з quantity = -1

**ER:** Очікувано 4xx. Якщо 200 — дефект: відсутня валідація


*Реальна поведінка ендпоінта інколи не відповідає Очікуваному Результату, але при написанні тест-кейсів я більше намагався передати корректну поведінку з моєї точки зору
